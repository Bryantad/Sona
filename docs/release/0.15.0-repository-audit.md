# Sona 0.15.0 Repository Audit

## Current Version

Sona is currently at `0.14.1`.

Baseline commands recorded on branch `feature/sona-0.15.0`:

| Command | Result |
| --- | --- |
| `python -m pytest -q tests` | Passed: 15 tests, 2 warnings, 85.72s |
| `python -m sona --version` | Passed: `Sona 0.14.1` |
| `python -m sona build-info` | Passed, but stdlib discovery path took about 29s |
| `python tools/run_examples.py` | Passed: 9 examples |
| `python -m sona probe stdlib` | Passed, median 30.449s across 3 runs |

The slow stdlib probe/build-info path is a 0.15.0 release concern and must be
converted to static manifest discovery.

## Repository Layout

- `sona/`: Python runtime, CLI, parser, interpreter, LSP, SPM, stdlib native modules.
- `stdlib/`: public `.smod` source modules shipped with the package.
- `tests/`: current regression tests.
- `examples/`: source-checkout examples.
- `docs/`: user-facing documentation and release notes.
- `scripts/`: release gates and legacy helper scripts.
- `tools/`: example runner and future release validation tools.

The primary `vscode-extension/` and `extensions/sona-receipt-explorer/`
directories are absent in this checkout because they were removed before the
0.14.1 release commit. VSIX work is therefore out of scope for 0.15.0 unless
those trees return as canonical source.

## Parser and Runtime Architecture

- Parser grammar lives in `sona/grammar.lark`.
- Main parser/runtime logic lives in `sona/parser_v090.py`,
  `sona/ast_nodes.py`, and `sona/interpreter.py`.
- `SonaUnifiedInterpreter` owns builtins, module loading, cognitive helper
  hooks, and execution.
- `.smod` modules are executed by `SimpleModuleSystem._load_smod_module`.
- Public imports flow through `SimpleModuleSystem.import_module`.
- `sona/__init__.py` uses lazy exports and no longer eagerly constructs a
  default interpreter at import time.

The grammar already supports a standalone `@intent` statement, but not
function-attached annotations. Guardian must not depend on parser changes.

## Standard Library Architecture

- Public `.smod` modules live in root `stdlib/`.
- Python stdlib/native compatibility modules live in `sona/stdlib/`.
- Manifest metadata lives in `sona/stdlib/MANIFEST.json`.
- Root `.smod` files are packaged through `pyproject.toml` and `MANIFEST.in`.
- Current source markers include `sona`, `sona+intrinsic`, and `intrinsic`.

For `source: "sona"` modules, the loader uses a null native bridge. For
`source: "sona+intrinsic"`, the loader exposes private intrinsics from
`sona/stdlib/native_intrinsics.py`.

## Public Modules

The 0.14.1 manifest currently reports 86 modules. The Sona-native foundation
modules include `queue`, `stack`, `sort`, `search`, `statistics`, `matrix`,
`graph`, and `permissions`. Sona-authored wrappers over intrinsics include
`hashing`, `random`, `uuid`, `secrets`, `password`, `jwt`, and `crypto`.

Some 0.15.0 utility surfaces already exist in partial form:

- `path.smod` exists but lacks the full requested `stem` and `relative` API.
- `csv.smod` exists but uses existing option-map APIs instead of the requested
  `read` and `write` names.
- `uuid.smod` exists and is backed by private intrinsics.
- `color` and `assert` currently exist as Python-backed preview modules.

Most 0.15.0 accessibility and Guardian modules do not exist yet.

## Hidden Native Modules

Hidden modules include `native_*`, `intrinsics`, `native_intrinsics`, and
`native_bridge`. The interpreter blocks direct user imports for these names,
and CLI/LSP/SPM filtering already hides the main private names. 0.15.0 must
extend this filtering to the canonical manifest and any new internal modules.

## Packaging Rules

- Package metadata is in `pyproject.toml`.
- `MANIFEST.in` includes `sona` grammar files and root `stdlib` `.smod` files.
- `MANIFEST.in` prunes repository-only content such as docs, examples, tests,
  extensions, build output, and VSIX artifacts.
- `scripts/release_hardening.ps1` currently builds wheel and sdist, inspects
  archives, installs a wheel in a temporary venv, and runs smoke checks from an
  external workdir.

0.15.0 must extend hardening to validate both wheel and sdist installation from
outside the repository and must run Guardian fixture smoke checks outside the
source tree.

## Release Hardening

Current hardening is Python-artifact only because VSIX source is absent. It
calls the fast release gate, builds wheel/sdist with `python -m build`, inspects
runtime packaging content, performs a wheel install smoke test from
`build/release-hardening/<version>/smoke-workdir`, and prints SHA-256 hashes.

0.15.0 must add:

- release metadata validation,
- static manifest validation,
- fast stdlib probe timing,
- accessibility and Guardian probes,
- clean-install checks for both wheel and sdist,
- Guardian fixture smoke checks outside the repository root.

## Existing Tests

Current tests are concentrated in `tests/test_sona_native_stdlib_0141.py`.
Baseline result is 15 passed. New 0.15.0 tests should be organized under
`tests/stdlib/`, `tests/accessibility/`, `tests/guardian/`, `tests/cli/`,
`tests/packaging/`, `tests/regression/`, and `tests/release/`.

## Known Defects

- `python -m sona probe stdlib` currently imports/checks modules and takes about
  30.449s median across three runs.
- `python -m sona build-info` includes the same slow stdlib discovery path.
- `date` lacks the exact requested `now`, `utcnow`, and `fromtimestamp` API
  spellings.
- `path.smod` and `csv.smod` need 0.15.0 API completion while preserving
  existing names.
- `fs.smod` is import-side-effect free, but its comments understate native
  exception behavior and need regression coverage.
- `stdin.smod` and `web.py` are not present in this checkout; no fake modules
  should be introduced for them.

## Migration Plan

1. Keep 0.14.1 behavior passing while adding 0.15.0 features.
2. Make the manifest the canonical source for discovery, docs validation,
   packaging validation, and release inventory.
3. Convert stdlib probing to static manifest reads before adding large module
   inventory.
4. Implement stable utility modules and shared cognitive internals before
   accessibility surfaces.
5. Implement Guardian with strict project-root confinement and safe default
   detect/report behavior.
6. Add all 55 cognitive-accessibility modules with tests and documentation.
7. Harden packaging and clean-install validation before release readiness.
