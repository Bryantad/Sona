# Sona v0.11.0 — Native `.smod` Runtime and Trust-Layer Hardening

Release date: 2026-03-03

Sona v0.11.0 consolidates the trust-layer reliability baseline (previously scoped as v0.10.4) with trust-layer hardening, native `.smod` architecture completion, and new runtime modules. This release delivers the governance baseline, receipt chain verification and signing/redaction, the replay contract system, the emotion module, and full `.smod` contract coverage — bringing Sona's module format from ad-hoc wrappers to a production-grade native runtime.

> **Note:** v0.10.4 was absorbed into v0.11.0. All features originally developed under the v0.10.4 cycle are included here.

## Headline Changes

- **Governance baseline** (from v0.10.4 cycle):
  - `sona receipt verify-chain` — chain integrity verification at rest
  - governance anchors in receipt headers: `policy_fingerprint`, `engine_policy_fingerprint`, `prev_receipt_hash`, `receipt_hash`
  - restored `sona.policy` (`sona probe`) and `sona.lockfile_manager` (`sona lock` / `sona verify`)
  - cross-platform `trust-layer-smoke` CI matrix (Windows/Linux/macOS)
  - spec draft baselines: `DETERMINISM_DRAFT_1.md`, `RECEIPT_SPEC_0.3_DRAFT.md`
- **Native `.smod` architecture completed:**
  - 18 total `.smod` modules (16 stdlib + `receipt.smod` + `emotion.smod`)
  - all explicit with `module_format` / `runtime_backend` metadata — zero modules in backlog
  - `NativeBridge` architecture for `smod-bridge` modules, pure-Sona execution for `smod` modules
  - `.smod` first-class in package manager (`spm.py`)
- Added signed-receipt baseline with CLI support:
  - `sona receipt sign`
  - `sona receipt verify-signature`
- Added HMAC-SHA256 signature envelope in receipt headers.
- Added deterministic receipt redaction baseline:
  - `sona receipt redact`
  - `sona receipt redact-dir`
  - profile-driven transforms (`dev`, `ci`, `prod`)
- Trust-layer ledger enhancements:
  - enriched chain certificates with integrity scores and signature coverage
  - portable receipt bundle export (`sona receipt export`)
  - replay contract system for deterministic drift detection
- Added receipt execution context APIs and run-path pre-event ingestion for richer provenance timelines.
- Added emotion module stack (`emotion.smod` + native bridge + backend) for state detection, composition, and response strategy mapping.
- Policy version promoted from `0.1-draft` to `0.1`.
- All 16 stdlib `.smod` modules now declare explicit `module_format` and `runtime_backend` metadata.

- Added `enrich_chain_certificate()` for integrity scoring (0.0–1.0), signature coverage analysis, and chain anchor extraction.
- Added `export_receipt_bundle()` for portable JSON bundles with chain verification, receipts, policy snapshot, and bundle hash.
- Added `ReplayContract` dataclass for deterministic replay expectations.

- Added `append_receipt_event()` to capture run-time provenance events into active receipt context.
- Extended `build_receipt()` with `pre_events` support to persist pre-execution and in-run events.
- Wired CLI run flow to create receipt context before execution and pass harvested pre-events into final receipt generation.

### Native `.smod` Runtime — Architecture and Completion

Sona `.smod` files are the native module format — Sona-syntax source files with explicit runtime metadata executed by the interpreter.

- **Two runtime categories:**
  - Pure-Sona (`runtime_backend = "smod"`): `queue`, `stack`, `random`, `uuid`, `regex`
  - Bridge (`runtime_backend = "smod-bridge"`): `hashing`, `env`, `fs`, `io`, `json`, `path`, `csv`, `date`, `math`, `string`, `time`, `receipt`, `emotion`
- **Loading pipeline:** `_resolve_smod_path()` → `_load_smod_module()` → `import_module()` resolving `.sona_modules/` (local) → `stdlib/` (built-in).
- **NativeBridge** (`native_bridge.py`): dynamically loads `native_<name>.py` with `NativeModuleProxy` prefix fallback.
- **Package manager** (`spm.py`): `.smod` files are first-class install targets; single `.smod` copy or `__init__.smod` for packages.
- **Metadata contract completion by wave:**
  - Wave-1 (v0.10.4 cycle): `json`, `fs`, `io`, `path`, `env`
  - Wave-2 (pre-v0.10.4): `queue`, `stack`, `random`, `uuid`, `regex` (pure), `hashing` (bridge)
  - Wave-3 (v0.11.0): `csv`, `date`, `math`, `string`, `time` — backlog cleared to zero
- Added explicit `module_format` and `runtime_backend` metadata to Wave-3 bridge modules.
- All 16 stdlib `.smod` files now declare `module_format = "smod-runtime"` with correct `runtime_backend`.
- Sub-module: `stdlib/utils/convert.smod`.
- Metadata contract test coverage expanded to 3 waves + exhaustive scan.
