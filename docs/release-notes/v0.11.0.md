# Sona v0.11.0 ‚Äî Native `.smod` Runtime and Trust-Layer Hardening

**Release tag:** `v0.11.0`  
**Date:** 2026-03-03  
**Status:** Complete

Sona v0.11.0 consolidates the trust-layer reliability baseline (previously scoped as v0.10.4) with trust-layer hardening, native `.smod` architecture completion, and new runtime modules. This release delivers receipt chain verification, governance anchors, receipt signing/redaction, the replay contract system, the emotion module, and the first fully explicit `.smod` contract surface ‚Äî bringing Sona's module format from ad-hoc wrappers to a production-grade native runtime.

> **Note:** v0.10.4 was absorbed into v0.11.0. All features originally planned and implemented under the v0.10.4 tag are included here.

### Native `.smod` Architecture

Sona `.smod` files are the native module format ‚Äî Sona-syntax source files executed by the interpreter with explicit runtime metadata contracts. This release completes the architecture end-to-end.

**Module format:**

- Each `.smod` file declares `const module_format = "smod-runtime"` and `const runtime_backend = "smod"` (pure Sona) or `"smod-bridge"` (requires Python native glue via `__native__` calls).

**Two runtime categories:**

- **Pure native** (`runtime_backend = "smod"`): `queue`, `stack`, `random`, `uuid`, `regex` ‚Äî 100% Sona code, no Python bridge required.
- **Bridge** (`runtime_backend = "smod-bridge"`): `hashing`, `env`, `fs`, `io`, `json`, `path`, `csv`, `date`, `math`, `string`, `time`, `receipt`, `emotion` ‚Äî use `__native__.*` calls routed through Python backends via `NativeBridge`.

**Loading pipeline** (in `sona/interpreter.py`):

- `_resolve_smod_path()` ‚Äî resolves to `.sona_modules/<name>.smod` (local packages) or `stdlib/<name>.smod` (built-in stdlib).
- `_load_smod_module()` ‚Äî strips comments, detects `__native__` usage, creates `NativeBridge` if needed, executes Sona source in isolated scope, and exports public symbols.
- `_load_module()` / `import_module()` ‚Äî handles `.smod` suffix stripping, `force_smod` flag, and fallback chain from `.smod` ‚Üí `native_*.py` ‚Üí `*.py`.

**NativeBridge** (`sona/stdlib/native_bridge.py`):

- `NativeBridge` class dynamically loads `sona.stdlib.native_<name>` or `sona.stdlib.<name>`.
- `NativeModuleProxy` exposes attributes with prefix fallback (`<module>_<attr>` ‚Üí `<attr>`).

**Package manager integration** (`sona/spm.py`):

- `.smod` files are first-class install targets.
- Single `.smod` files copy directly to `.sona_modules/`.
- Package directories use `__init__.smod` entry points.

### Receipt Signing and Redaction

- `sona receipt sign`
- `sona receipt verify-signature`
- Added deterministic redaction commands:
  - `sona receipt redact`
  - `sona receipt redact-dir`
- Added receipt signing primitives in `sona/receipts.py`:
  - canonical signable payload generation
  - HMAC-SHA256 signature generation
  - signature verification with `key_id` checking support
- Added deterministic redaction primitives in `sona/receipts.py`:
  - profile-driven policy resolution (`dev`, `ci`, `prod`)
  - tier-based classification (`public`, `internal`, `sensitive`)
  - deterministic hashed redaction tokens (`[REDACTED:sha256:<hash>]`)
  - redaction metadata in `header.redaction`
- Added draft spec: `docs/spec/RECEIPT_REDACTION_SPEC_0.1_DRAFT.md`
- Signature metadata now stored in receipt header as `header.signature`.
- Receipt hashing remains stable across signed/unsigned forms by excluding signature envelope fields from canonical hash payload.
- Receipt hashing remains deterministic for redacted receipts with stable canonical payload rules.
